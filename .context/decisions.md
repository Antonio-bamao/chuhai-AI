# 决策记录

> 记录高影响决策，不要只记录结论，要写背景、取舍和后续约束。决策只有利弊，没有绝对对错；不写放弃了什么，就不允许记录。

## ADR-0001

- 编号：ADR-0001
- 标题：去授权采用“鉴权点短路返值”而非删除登录模块
- 状态：accepted
- 日期：2026-06-20
- 背景（Context）：客户端存在 ProGuard 混淆和字符串/资源加密，登录、套餐、鉴权与启动流程耦合；服务器不可动，业务联网和 native 签名链路必须保留。
- 决策（Decision）：定位唯一“鉴权结果判定点”，让它恒返 ok，并提供永久或足够远的 `expireAt` 与全解锁 `plan`；不删除登录模块，不跨模块大改。
- 取舍（利弊，放弃了什么）：获得最小改动面、可回滚性、静默性和对业务链路的低影响；放弃快速删除 UI、封域名、全量重写等表面更直接但风险更高的做法。代价是必须先完成字符串解密和接缝定位，前期分析成本更高。
- 后果（Consequences）：后续所有去授权改动必须围绕 `design.md` 第 3 节的授权握手接缝；若需要改变接缝契约，必须新增 ADR 并经用户批准。
- 取代关系（Supersedes）：无
